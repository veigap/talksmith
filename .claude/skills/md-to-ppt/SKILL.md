# md-to-ppt — Render `master.md` to PowerPoint

## When to use

After the Review loop (Step 6) closes and the presenter declares `master.md` final. This is **optional** — many presenters will stop at the outline and render slides with their own tooling.

The skill operates on the active `Talk`:

```
talks/<Talk>/
├── master.md                 # source of truth — input to this skill
└── output/
    └── <Talk>.pptx           # generated by this skill
```

## Inputs

- **Active `Talk` path** — must be passed in. If missing, stop and ask.
- **`profile.md` content** — the orchestrator will include the global presenter profile when non-empty. Respect any rendering-relevant preferences (e.g. tone, no-stock-photos, no thank-you slide, layout preferences).
- **Template** — default to [`templates/template.pptx`](../../../templates/template.pptx) at the repo root. This `.pptx` file defines the master slide layouts (title, content, two-column, divider, etc.) and the deck's theme (fonts, colors, logo). Only deviate from this default if the presenter explicitly passes a different template path. If `templates/template.pptx` is missing, stop and ask before generating anything.

## Process

1. Confirm the active `Talk` path and locate `master.md`.
2. Resolve the template path (default `templates/template.pptx`) and confirm it exists.
3. Parse `master.md`:
   - Frontmatter → title slide metadata (presenter, audience, duration, date).
   - `Thesis` → subtitle on title slide.
   - `Agenda` → agenda slide listing Sections.
   - For each Section:
     - One divider slide (Section title + "Goal of this section").
     - For each Slide inside the Section: one content slide using `Content` as the body and `Speaker notes` as the slide's speaker notes field.
   - `Conclusions` → closing slide(s).
   - `Open questions` / `Cut material` → **do not include** in the deck. They are working notes.
4. Map each slide to a layout from the template (title, section header, content, two-column, etc.). When in doubt, default to a plain content layout.
5. Write the deck to `talks/<Talk>/output/<folder-name>.pptx`.
6. Report what was generated: slide count, layouts used, anything that didn't map cleanly.

## Implementation notes

- Use `python-pptx` (preferred) to render the deck against the supplied template — it preserves master slide layouts and theme.
- Do **not** invent layouts; only use layouts defined in the template.
- Speaker notes go into the slide's notes pane, not on the slide body.
- Cite-by-filename references in `Sources` (e.g. `compile/foo.md`) can be appended as a small footer or dropped — confirm with the presenter on the first run.

## Failure modes to surface

- Template path missing or unreadable.
- `master.md` not yet filled in (frontmatter/Thesis/Agenda empty).
- A Section has zero Slides.
- A slide's `Content` exceeds reasonable length for the chosen layout — flag it instead of truncating.
